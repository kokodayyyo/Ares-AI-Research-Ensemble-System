# First Principles of Heuristic Design
When designing heuristic algorithms, strictly apply these 4 pillars:
1. **Bounded Rationality (Goal):** Sacrifice absolute optimality for speed. Fight combinatorial explosion by seeking good-enough solutions with cheap evaluations.
2. **Representation & Proxies (Compass):** Embed domain knowledge through effective solution representations, neighborhoods, and lightweight information proxies.
3. **Dynamic Balance (Engine):** Adaptively balance exploration and exploitation through controlled randomness and feedback-driven adjustment over time.
4. **Robustness (Constraint):** Prefer stable designs over brittle tuning. Aim for methods that generalize across diverse instances rather than overfitting to narrow cases.

# Strategy Table Rules
Build the initial strategy table under these rules:
1. Factors must be implementable mechanisms, not broad slogans.
2. Core drivers should receive higher initial weights, and utility tools should receive lower weights.
3. Assign initial weights as follows:
   - Core mechanisms that directly implement strong proxies, breakout search dynamics, or self-regulating adaptation should usually start at `0.6 to 0.8`.
   - Supporting or utility strategies may appear, but should usually remain at `0.4 to 0.5`.
4. Each factor must include one concrete role/hypothesis and one concrete pitfall.
5. The table must be easy to translate into code.
6. Do not use generic factor names such as "exploration", "optimization", or "adaptation" unless they are grounded in a specific mechanism.
7. If a classic heuristic is included, treat it as a supporting utility tool unless it is clearly central to the current problem structure.
