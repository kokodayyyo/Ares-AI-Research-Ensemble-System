# A User Guide to the ARES System

ARES (AI Research Ensemble System) is an automated system for algorithm evolution and discovery driven by large language models (LLMs). Through multi-role collaboration among the Theorist, Critic, and Experimenter, it automatically generates, validates, and optimizes code solutions for specific problems.

Based on the current implementation, `ares new2.py`, this document summarizes the actual execution workflow of the system, its evaluation logic, the requirements for integrating new problems, and the key configuration items.

## 1. What ARES Actually Does

ARES is not a one-shot code generator. In the current implementation, it operates as a multi-stage evolutionary closed loop:

1. The Theorist generates an initial Strategy Table from the initialization prompt.
    
2. The system evaluates the seed function as the initial elitist reference.
    
3. The Experimenter generates the initial candidate population based on the initial Strategy Table.
    
4. All candidate programs are automatically executed and scored in isolated temporary directories.
    
5. In each subsequent iteration:
    
    - The Theorist updates a hypothetical strategy table and proposes atomic validation experiments;
        
    - The Critic validates and debugs these proposals through structural ablation (via LLM) or parameter scanning (via local scripts);
        
    - The Theorist integrates the validation evidence into a final guide and a new strategy table;
        
    - The Experimenter generates the next generation of offspring based on the updated strategy table and final guide.
        
6. ARES retains the best-performing individuals through elitist survivor selection.
    
7. When the system detects stagnation, it triggers Radical Exploration and periodically performs Meta-Reflection.

## 2. Runtime Outputs and Working Directory

Each run creates a timestamped working directory:

```text
LLM-outputs/<YYYY-MM-DD_HH-MM-SS>/
```

This directory stores the outputs produced by different roles:

- `theorist_outputs/`: hypothetical tables, proposals, final guides, and related artifacts
    
- `critic_outputs/`: mutants generated by the Critic, validation reports, parameter scan logs, and related outputs
    
- `experimenter_outputs/`: responses generated by the Experimenter, code files, and stdout logs
    
- `weights_history.txt`: the historical record of Strategy Tables across iterations
    

This structure ensures traceability for each run and facilitates debugging and post hoc analysis.

## 3. Core System Configuration (`cfg/config.yaml`)

System behavior is primarily controlled by `cfg/config.yaml`.

### 3.1 Basic Runtime Parameters

|Parameter|Meaning|
|:--|:--|
|`num_eval_workers`|Number of processes used for parallel evaluation|
|`timeout`|Timeout for a single candidate evaluation, in seconds|
|`eval_runs`|Number of repeated executions per candidate. An individual is considered successful only if all runs complete successfully and return valid scores|
|`init_pop_size`|Size of the initial population generated during initialization|
|`pop_size`|Number of offspring generated by the Experimenter in each evolutionary iteration|
|`max_fe`|Maximum cumulative number of offspring evaluations in the main evolutionary loop|
|`stagnation_threshold`|Minimum improvement threshold for determining whether meaningful progress has been made|

One detail deserves special clarification. In the current implementation, `function_evals` is incremented only for newly generated `evaluated_offspring` within the main loop. It does not include evaluations of the seed, individuals generated during initialization, or candidates produced during Critic validation. Therefore, `max_fe` should be understood more precisely as the evaluation budget for main-loop offspring, rather than the total execution budget for all individuals in the entire system.

### 3.2 Advanced Reasoning and Validation Parameters

|Parameter|Meaning|
|:--|:--|
|`meta_reflection_cycle_period`|Iteration interval for periodically triggering Meta-Reflection|
|`radical_exploration_trigger`|Number of consecutive stagnant iterations required to trigger Radical Exploration|
|`radical_exploration_duration`|Number of iterations for which Radical Exploration remains active|

### 3.3 Problem and Role Configuration

The current implementation requires the following problem- and role-related fields in the configuration:

- `cfg.problem.problem_name`
    
- `cfg.problem.description`
    
- `cfg.problem.func_name`
    
- `cfg.problem.obj_type` (`min` or `max`)
    
- `cfg.llm_roles.theorist`
    
- `cfg.llm_roles.critic`
    
- `cfg.llm_roles.experimenter`
    

The main entry program also reads:

- `cfg.algorithm`
    

to record the name of the algorithm/framework currently in use.

## 4. Evaluation Logic and Failure Handling

This part is critical, as it determines whether a candidate program can actually survive within the system.

### 4.1 Isolated Evaluation

Each candidate program is evaluated in a fresh temporary directory. ARES performs the following steps:

1. It copies the entire problem directory under `problems/<problem_name>/`;
    
2. It writes the current candidate code to `gpt.py`;
    
3. It executes `eval.py` within the isolated directory;
    
4. It writes stdout/stderr to log files;
    
5. A Python process pool is used here to accelerate execution.
    

This design prevents contamination across individuals and preserves the integrity of the original benchmark files.

### 4.2 Score Parsing Rule

ARES searches stdout in reverse order and takes the last line that can be parsed as a float as the fitness score.

Therefore, `eval.py` should ideally print the final score clearly on the last line of standard output.

### 4.3 Conditions Under Which an Individual Is Considered Failed

A candidate program is classified as a failed individual if any of the following occurs:

- execution timeout
    
- Python exception
    
- missing final score or inability to parse the score as a numeric value
    
- failure of any run among repeated evaluations
    

Failed individuals are assigned:

- `exec_success = False`
    
- `obj = inf`
    

Thus, in the current implementation, a code sample must satisfy both executability and stability under repeated evaluation in order to enter the evolutionary selection process.

## 5. Multi-Agent Evolutionary Workflow

ARES operates through collaboration among the Theorist, Critic, and Experimenter. During initialization, the Theorist generates the initial Strategy Table, the system evaluates the seed function, and the initial population is generated accordingly.  
This step is based on `prompts/<problem_name>/init.txt`, whose content is written by the user according to the target objective function. It is injected only once during initialization and is used solely to assist in generating V0.

In subsequent iterations, the Theorist first updates its strategic understanding and proposes experiments to be validated. The Critic then performs validation through structural ablation or parameter scanning. If the Critic discovers a superior individual, the system adopts it immediately within the current iteration. The Theorist then synthesizes the proposals, validation results, and code evidence to produce a final guide and, when formatting requirements are satisfied, updates the central Strategy Table. The Experimenter subsequently generates the next generation of code based on this guide. The system then completes evaluation, deduplication, and elitist retention to form the new population.

When the search stagnates for multiple consecutive iterations, ARES triggers Radical Exploration. The system first saves a checkpoint of the current stable state, then enters several iterations of high-risk exploration. If the exploration yields results superior to the original checkpoint, the new path is retained; otherwise, the system rolls back. This design makes radical exploration a controlled mechanism rather than an irreversible random jump.

## 6. Problem Definition and Required Files

To integrate a new problem, one must provide both the benchmark evaluation logic and the prompt context.

### Step 1: Benchmark Directory (`problems/`)

Create a new folder under `problems/` with the same name as the problem, for example:

```text
problems/my_new_task/
```

This directory should contain at least:

- `gpt.py`  
    the target file into which candidate code is written. During evaluation, ARES overwrites it with the generated code.
    
- `eval.py`  
    the problem evaluation script. It should execute the candidate logic defined in `gpt.py` and output the final fitness score.
    

Additional auxiliary files may also be placed in this directory. Since ARES copies the entire problem directory during evaluation, dependencies on extra resource files are permitted.

### Step 2: Prompt Context Directory (`prompts/<problem_name>/`)

Create a folder under `prompts/` with the same name as the problem directory, for example:

```text
prompts/my_new_task/
```

In the current implementation, the system directly reads the following problem-level prompt files:

1. `func_signature.txt`: function signature and input/output interface requirements
    
2. `func_desc.txt`: problem description (optional)
    
3. `seed_func.txt`: a valid seed function, used directly as the seed code during initialization
    
4. `init.txt`: an optimization template used only during initialization for the Theorist to generate V0; its content is similar to that of `func_signature.txt`
    

Among these, the first three files are read by the system, whereas `init.txt` is injected through `{description}` into the Theorist’s initialization prompt only within `init_population()`. It is used once for V0 generation and is not reused in subsequent iterations.

### Step 3: Problem Configuration (`cfg/problem/`)

Create a corresponding configuration file under `cfg/problem/`, for example:

```text
cfg/problem/my_new_task.yaml
```

It should contain at least the following fields:

- `problem_name`
    
- `problem_type`
    
- `func_name`
    
- `obj_type`
    

### Step 4: Register in the Main Configuration

Finally, complete the integration in the main configuration file `cfg/config.yaml` by:

- pointing the current problem to the newly created problem configuration
    
- configuring the LLM clients for each role
    
- setting the evolutionary parameters according to the characteristics of the task

## 8. Global Prompt Templates for ARES

In addition to the problem-level prompt files, the current implementation also requires ARES-level template files under:

```text
prompts/ares/
```

These files include:

- `user_generator_theorist_init.txt`
    
- `user_reflector_theorist_p1_task1_update_table.txt`
    
- `user_reflector_theorist_p1_task2_propose_innovation.txt`
    
- `user_reflector_theorist_phase2_conventional.txt`
    
- `user_reflector_theorist_radical.txt`
    
- `user_reflector_theorist_meta.txt`
    
- `user_reflector_critic_ablation.txt`
    
- `user_reflector_critic_scan.txt`
    
- `user_reflector_critic_mixed.txt`
    
- `user_generator_experimenter.txt`
    
- `ares_strategy_table_template.md`
    

If any of these files are missing, the workflow cannot be executed completely.

In addition, if the problem-level `init.txt` is enabled, then `user_generator_theorist_init.txt` should contain the `{description}` placeholder to receive the initialization-specific description.

## 9. LLM Interface Configuration

ARES supports assigning different LLM backends to different roles. In the current implementation, the main program instantiates three role-specific clients separately:

- Theorist LLM
    
- Critic LLM
    
- Experimenter LLM
    

These clients are created through `cfg.llm_roles.*` using Hydra.

This means that one may:

- assign different models to different roles
    
- specify different temperature settings for different roles
    
- make the Critic more deterministic while allowing the Experimenter to be more exploratory
    

---

## 10. Quick Start

1. Configure the API key in `cfg/llm_client`.
    
2. Prepare the benchmark directory under `problems/<problem_name>/`, containing at least `gpt.py` and `eval.py`.
    
3. Prepare the problem-level prompt files under `prompts/<problem_name>/`: `func_signature.txt`, `func_desc.txt`, `seed_func.txt`, and `init.txt`.
    
4. Ensure that the global ARES template files under `prompts/ares/` are complete and available.
    
5. Configure the corresponding YAML file for the problem under `cfg/problem/`, and set the current `problem_name`, role-specific LLMs, and evolutionary parameters in `cfg/config.yaml`.
    
6. Run the main program to launch ARES.
    

The repository also provides execution scripts for HGS and LKH, which may serve as reference baselines for specific problems. Running the corresponding scripts in the `USE` folder will solve the associated datasets.
